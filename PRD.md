# PRD — 반도체 시료 생산주문관리 시스템 (SimpleOrderSystem)

> `docs/01-개요.md` ~ `docs/09-미션및제출방법.md`의 기능 명세를 통합 정리한 제품 요구사항 문서다. 상세 화면 예시와 배경은 각 `docs/` 파일을 참고하고, 이 문서는 "무엇을 만들어야 하는가"를 한곳에서 확인하기 위한 요약이다.

## 1. 배경 및 목표

가상의 반도체 회사 "S-Semi"는 반도체 시료(Sample)를 생산해 고객(연구소/팹리스/대학 연구실)에 납품한다. 주문량 증가로 엑셀/메모장 기반 관리가 한계에 부딪혀, 아래를 해결하는 콘솔 기반 생산주문관리 시스템을 개발한다.

- 주문 처리 현황을 한눈에 파악할 수 있어야 한다.
- 공정 예약 후 완성 시점을 추정할 수 있어야 한다.
- 재고가 충분한데 불필요한 추가 공정이 돌아가지 않아야 한다.

## 2. 이해관계자 역할

| 역할 | 책임 |
|---|---|
| 고객 | 시료 요청(이메일, 시스템 외부) |
| 주문 담당자 | 요청에 맞게 주문서 작성(시료 주문 메뉴) |
| 생산 담당자 | 시료 등록, 주문 승인/거절, 생산·출고 처리 |

## 3. 도메인 모델

| 엔티티 | 필드 |
|---|---|
| Sample(시료) | 시료 ID, 이름, 평균 생산시간(min/ea), 수율(0 초과 1 이하), 현재 재고 |
| Order(주문) | 주문번호(`ORD-YYYYMMDD-NNNN`), 시료 ID, 고객명, 주문 수량, 상태, 생성/변경 시각, (승인 시) 부족분/실생산량/총생산시간 |

### 주문 상태 흐름

```
RESERVED --거절--> REJECTED
RESERVED --승인, 재고 충분--> CONFIRMED --출고--> RELEASE
RESERVED --승인, 재고 부족--> PRODUCING --생산완료--> CONFIRMED --출고--> RELEASE
```

`REJECTED`는 정상 흐름 밖의 상태이며 모니터링/생산라인 집계에서 제외한다.

## 4. 기능 요구사항

### 4.1 메인 메뉴 (`docs/02`)
시스템 현황(현재 일시, 등록 시료 종수, 총 재고, 전체 주문 건수, 생산라인 대기 건수)을 표시하고 `[1]`~`[6]`, `[0]` 메뉴로 분기한다.

### 4.2 시료 관리 (`docs/03`)
- 등록: 시료 ID, 이름, 평균 생산시간, 수율 입력.
- 조회: 등록된 전체 시료 + 현재 재고 표시.
- 검색: 이름으로 검색.

### 4.3 시료 주문 (`docs/04`)
시료 ID, 고객명, 주문 수량을 입력받아 `RESERVED` 상태의 주문을 생성한다. 채번 규칙: `ORD-YYYYMMDD-NNNN`(같은 날짜 내 순번).

### 4.4 주문 승인/거절 (`docs/05`)
`RESERVED` 목록을 표시하고, 특정 주문에 대해:
- **승인**: 재고 ≥ 주문 수량이면 즉시 `CONFIRMED`(재고 차감). 부족하면 부족분/실생산량/총생산시간을 계산해 `PRODUCING`으로 전환(생산 큐 등록).
- **거절**: 즉시 `REJECTED`로 전환.

### 4.5 모니터링 (`docs/06`)
- 주문량 확인: 상태별(`RESERVED`/`CONFIRMED`/`PRODUCING`/`RELEASE`) 건수(`REJECTED` 제외).
- 재고량 확인: 시료별 재고 + 상태(여유/부족/고갈).

### 4.6 생산 라인 (`docs/07`)
단일 생산 라인 + FIFO 큐. `PRODUCING` 주문을 큐잉 시각(승인 시각) 순으로 누적 스케줄링하여 각 주문의 진행률(%)과 예상 완료 시각을 계산·표시한다. 완료된 주문은 실행 시점(재실행 포함)에 실제 경과 시간을 기준으로 `CONFIRMED`로 전환하고 재고를 갱신한다.

### 4.7 출고 처리 (`docs/08`)
`CONFIRMED` 주문 목록을 표시하고 선택한 주문을 `RELEASE`로 전환한다.

## 5. 계산 공식 (반드시 준수)

- 부족분 = 주문 수량 − 현재 재고
- 실 생산량 = `ceil(부족분 / 수율)`
- 총 생산 시간(분) = 평균 생산시간 × 실 생산량
- 생산 완료 시 재고에는 실 생산량 **전체**(수율 반영분 포함)가 반영된다.

## 6. 비기능 요구사항

- **데이터 영속성**: JSON 파일(`data/samples.json`, `data/orders.json`). 애플리케이션 재실행 후에도 상태가 유지되어야 한다.
- **실시간 반영**: 백그라운드 동작은 아니지만, 재실행 시점에 실제 경과 시간을 확인해 완료된 생산을 반영해야 한다(`CatchUpProduction`).
- **인코딩**: 소스는 서명 있는 UTF-8(BOM)로 저장하고, 콘솔 출력도 실행 시점에 UTF-8 코드페이지(65001)로 강제해 한글이 어떤 시스템 로케일에서도 깨지지 않아야 한다.
- **테스트**: gtest/gmock 기반 TC를 Debug 빌드에서만 자동 수행하고, Release 바이너리에는 테스트 프레임워크가 포함되지 않는다.
- **더미 데이터**: `--seed` 옵션으로 상태별 분포를 갖춘 기본 데이터셋을 생성할 수 있어야 한다(수동 테스트/데모용).

## 7. 아키텍처 개요

MVC + 계층별 책임 분리(4개 에이전트 담당):

| 계층/모듈 | 책임 | 담당 |
|---|---|---|
| `Model/` | 공유 도메인 타입(`Sample`, `Order`, `OrderStatus`, `StockState`) | director(공유 계약) |
| `Persistence/` | `IDataStore` 인터페이스, `JsonDataStore`(JSON 파일 기반 CRUD) | persistent |
| `Monitor/` | `OrderService`(승인/거절/출고 상태 전이, 재고 계산, FIFO 생산 큐·진행률) | monitor |
| `Console/` | 메인 메뉴 + 6개 서브 메뉴 View/Controller | console |
| `Dummy/` | `DummyDataGenerator`(더미 데이터 생성/시드) | dummy |
| `Tests/` | gtest/gmock TC, 공유 `Tests/Mocks/MockDataStore.h` | persistent/monitor/dummy |
| `main.cpp` | 전체 조립, Debug TC 전용 실행, Release 콘솔 앱 실행 | director |

각 계층은 인터페이스(`IDataStore`, `OrderService`)를 통해서만 통신하며, 도메인 규칙(재고 계산·상태 전이)의 단일 진실 공급원은 `Monitor::OrderService`다.

## 8. 범위 밖 (Out of Scope)

- 사용자 인증/권한(로그인) — 역할은 개념적 구분일 뿐 시스템 내 계정 기능은 없다.
- 다중 생산 라인 — 현재는 단일 라인 + FIFO만 지원한다(문서 예시 기준).
- 네트워크/서버 통신 — 완전한 로컬 콘솔 애플리케이션이다.
